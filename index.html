Motivating Example

Some code is harder to test than others.   I spend a lot of time writing multithreaded and asynchronous code and worrying a lot about the purity of my interfaces.   The perennial Money class is a simple enough way to introduce a unit testing framework, but it doesn't look much like the code I have to test.

Let's instead begin our motivating example with something more complex.   Let's say we have to deliver a music database that aggregates data from various public Web Services.   While these examples still look a bit contrived, they immediately stress the capabilities of conventional Java unit testing frameworks like JUnit and TestNG.

Testing An Asynchronous HTTP Client

We'll start by building a convenience class on top of Apache's asynchronous HTTP client.   Given an array of URLs and a callback object, the convenience class will issue HTTP GET requests for each URL, parse any XML response bodies, and pass the parsed XML documents to a callback.   If any errors are encountered along the way, the client will pass exceptions instead of XML documents to the callback.

On top of that asynchronous method, we'll add a synchronous convenience wrapper.

Here is the code we have to test:
/**
 * A HTTP client that fetches and parses a XML response from n URLs concurrently and asynchronously.
 */
public class XmlFetcher {

    public XmlFetcher() {
    }

    public interface Callback {
        public void completed(Map<String, Object> results);
    }

    /**
     * GET and parse n XML responses from n URLs.
     *
     * @param URLs             The URLs to fetch
     * @param externalCallback A user-supplied callback that will be passed a map of URL to Object results.  Objects
     *                         can be downcast to either an org.w3c.dom.Document or a java.lang.Exception
     */
    public void fetch(final String[] URLs, final Callback externalCallback) {
        final Map<String, Object> results = new HashMap<String, Object>();
        final DefaultHttpAsyncClient client;

        try {
            client = new DefaultHttpAsyncClient();
        } catch (IOReactorException ex) {
            for (String url : URLs) {
                results.put(url, ex);
            }

            return;
        }

        client.start();

        for (final String URL : URLs) {
            client.execute(new HttpGet(URL), new FutureCallback<HttpResponse>() {
                private String url = URL;

                public void completed(HttpResponse response) {
                    if (200 > response.getStatusLine().getStatusCode() || 300 <= response.getStatusLine().getStatusCode()) {
                        handleEvent(new HttpException(response.getStatusLine().getReasonPhrase()));

                        return;
                    }

                    HttpEntity body = response.getEntity();

                    try {
                        DocumentBuilder docBuilder = DocumentBuilderFactory.newInstance().newDocumentBuilder();

                        Document document = docBuilder.parse(body.getContent());

                        handleEvent(document);
                    } catch (Exception ex) {
                        handleEvent(ex);
                    }
                }

                public void failed(Exception ex) {
                    handleEvent(ex);
                }

                public void cancelled() {
                    handleEvent(new CancellationException());
                }

                private void handleEvent(Object object) {
                    boolean isComplete = false;

                    synchronized (results) {
                        results.put(url, object);

                        if (results.size() == URLs.length) {
                            isComplete = true;
                        }
                    }

                    if (isComplete) {
                        // TODO - do a client.shutdown() without a deadlock.
                        externalCallback.completed(results);
                    }
                }
            });
        }
    }

    /**
     * GET and parse n XML responses from n URLs
     *
     * @param URLs The URLs to fetch
     * @return A map of URL to Object results.  Objects can be downcast to either an org.w3c.dom.Document or a java.lang.Exception
     * @throws InterruptedException
     */
    public Map<String, Object> fetch(String[] URLs) throws InterruptedException {
        final CountDownLatch latch = new CountDownLatch(1);
        final Map<String, Object>[] result = (Map<String, Object>[]) new Map[1];

        fetch(URLs, new Callback() {
            @Override
            public void completed(Map<String, Object> results) {
                result[0] = results;
                latch.countDown();
            }
        });

        latch.await();

        return result[0];
    }
}

A Simple Functional Test

How can we test this code?   We can start with a simple functional test using a conventional unit testing framework like TestNG:
@Test
public class XmlFetcherTest {
    private static final String URL = "http://musicbrainz.org/ws/1/artist?limit=1&query=madonna";

    public void functionalTest() throws Exception {
        XmlFetcher client = new XmlFetcher();

        client.fetch(new String[]{URL}, new XmlFetcher.Callback() {
            @Override
            public void completed(Map<String, Object> results) {
                Object result = results.get(URL);

                assert result instanceof Document;

                Document document = (Document) result;

                NodeList nodeList = document.getElementsByTagName("name");

                assert null != nodeList;
                assert 1 == nodeList.getLength();

                Node node = nodeList.item(0);

                assert "Madonna".equals(node.getTextContent());
            }
        });
    }
}
This test makes no modifications to the class under test's public interface.   It simply passes in a URL and does a minimal sanity check on the structure and content of the XML response.  

If you're like me, at this point you'll want to see the XML response.   Here it is:
<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<metadata xmlns="http://musicbrainz.org/ns/mmd-1.0#" xmlns:ext="http://musicbrainz.org/ns/ext-1.0#">
    <artist-list offset="0" count="13">
		<artist type="Person" id="79239441-bfd5-4981-a70c-55c3f15c1287" ext:score="100" >
			<name>Madonna</name>
			<sort-name>Madonna</sort-name>
			<life-span begin="1958-08-16"/>
		</artist>
	</artist-list>
</metadata>
The problem with this end to end style of testing is that it's brittle and it's difficult to isolate issues when they occur.   The Web Services provider, musicbrainz.org, might release a new version of their service that breaks backwards compatibility.   The content of the XML response could vary over time.   Intermittent network connectivity issues could also break our test.   In this case brittleness and fault isolation are related - there are so many ways that this test can break that it's difficult to isolate the problems when they occur.

If you're paying attention to code coverage, you'll also find that it's difficult to exercise the error handling parts of your source code.

Unit Testing With Mock Objects

We can address the brittleness and lack of isolation of the functional test by replacing our convenience class's real HTTP client with a mock object.   This mock object would return a canned XML response regardless of network conditions, etc.   If the unit test fails you can be sure that the failure is not due to some external, uncontrolled condition.

Here is our unit test again, this time with a mock object that returns a canned response:
@Test
public class XmlFetcherTest {
    private static final String URL = "http://musicbrainz.org/ws/1/artist?limit=1&query=madonna";
    private static final String XML = "<?xml version=\"1.0\" encoding=\"UTF-8\" standalone=\"yes\"?><metadata xmlns=\"http://musicbrainz.org/ns/mmd-1.0#\" xmlns:ext=\"http://musicbrainz.org/ns/ext-1.0#\"><artist-list offset=\"0\" count=\"13\"><artist type=\"Person\" id=\"79239441-bfd5-4981-a70c-55c3f15c1287\" ext:score=\"100\"><name>Madonna</name><sort-name>Madonna</sort-name><life-span begin=\"1958-08-16\"/></artist></artist-list></metadata>";

    public void functionalTest() throws Exception {
        ...
    }

    public void testValidResponseAsync() throws Exception {
        DefaultHttpAsyncClient mock = new DefaultHttpAsyncClient() {
            @Override
            public Future<HttpResponse> execute(HttpUriRequest request, FutureCallback<HttpResponse> callback) {
                try {
                    BasicHttpResponse response = new BasicHttpResponse(HttpVersion.HTTP_1_1, 200, "OK");

                    BasicHttpEntity entity = new BasicHttpEntity();

                    byte[] content = XML.getBytes("UTF-8");

                    entity.setContentLength(content.length);
                    entity.setContentType("application/xml;charset=\"utf8\"");
                    entity.setContent(new ByteArrayInputStream(content));

                    response.setEntity(entity);

                    callback.completed(response);
                } catch (Exception ex) {
                    callback.failed(ex);
                }

                return null;
            }
        };

        XmlFetcher client = new XmlFetcher(mock);

        client.fetch(new String[]{URL}, new XmlFetcher.Callback() {
            @Override
            public void completed(Map<String, Object> results) {
                Object result = results.get("http://musicbrainz.org/ws/1/artist?limit=1&query=madonna");

                assert result instanceof Document;

                Document document = (Document) result;

                NodeList nodeList = document.getElementsByTagName("name");

                assert null != nodeList;
                assert 1 == nodeList.getLength();

                Node node = nodeList.item(0);

                assert "Madonna".equals(node.getTextContent());
            }
        });
    }

    public void testValidResponseSync() throws Exception {
        ...
    }
}
Unfortunately this isn't nearly as straightforward as our functional test.    A couple of things are going on here:
  1. We take a real XML response and stuff it into a String.   We could have constructed this by hand, but capturing a real response saved a bit of time.
  2. Internally the class under test uses Apache's DefaultHttpAsyncClient as its HTTP client.   We subclass DefaultHttpAsyncClient with an anonymous class so it has the same type.   This is our mock object.   This mock object parses our XML string and passes it back to the callback.
  3. We 'injected' our mock object into the class under test so it calls our mock instead of hitting the wire with a real DefaultHttpAsyncClient
'Injected' is in quotes here because we're not using a real dependency injection framework like Spring or Guice.   Still, since most dependency injection frameworks rely exposing a class's dependencies in its public constructors or setters, this example should be sufficient to illustrate our next point.

Changing A Class In Order To Test It

Here comes the point.   In order to properly unit test our asynchronous HTTP client convenience class we had to change it.   Here's a before and after diff compliments of IntelliJ.   On the right is the original code.  On the left is the code modified to support injection of our mock object:



Here are the changes we made:
  1. We made the DefaultHttpAsyncClient local variable a private instance variable.
  2. We made the DefaultHttpAsyncClient local variable settable by exposing it in a new public constructor.   We could have just as easily exposed it in a public setter.
Simple enough, but what are the consequences?
  1. If our class is shared between threads, it may no longer be threadsafe.   Every invocation of fetch() will now use the same DefaultHttpAsyncClient instance variable where before it executed a distinct local variable.
  2. Implementational details of our class are now exposed in its public API.   The client code now knows that our class depends on both org.apache.http.impl.nio.client.DefaultHttpAsyncClient and org.apache.http.nio.reactor.IOReactorException.
  3. The client code now has to be a bit more complex than it used to.   In the original code IOReactorException was caught and handled locally.   In the new code, it's now exposed in the constructor so has to be handled by the client code.
You could argue that these consequences are worth it given the benefits of better fault localization and less brittleness in test cases, but it's a shame that we had to change our code this way in order to test it.

The Problem

Let's take a step back, look at the problem at a higher level, and see if we can derive a few requirements for a better solution.  

First we add a few more test cases to make this more realistic.   We add test cases to test handling of various error conditions - Network, HTTP, and XML errors.   Since our sync method is just a thin wrapper on top of our async fetch method, we only add a single sanity check test case for our sync method.

Next we add different mocks to simulate different error and success conditions.   Sometimes mocks can be shared between test cases, but sometimes it's easier to separate concerns and let mocks specialize.
